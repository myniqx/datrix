# Schema

Schemas are plain TypeScript objects that describe your data models. Forja uses them to
validate data, generate migrations, and build queries.

## Defining a schema

```typescript
import type { SchemaDefinition } from '@forja/types'

export const userSchema = {
  name: 'user',
  fields: {
    name:  { type: 'string', required: true },
    email: { type: 'string', required: true, unique: true },
    age:   { type: 'number' },
  },
} satisfies SchemaDefinition
```

The `satisfies` keyword gives full type checking while preserving the literal type ā€”
Forja needs this for type inference to work correctly.

## Reserved fields

Every schema automatically gets these fields ā€” you cannot define them manually:

| Field | Type | Description |
|-------|------|-------------|
| `id` | `number` | Auto-incremented primary key |
| `createdAt` | `date` | Set on record creation |
| `updatedAt` | `date` | Updated on every write |

## Schema options

```typescript
{
  name: 'user',           // Model name ā€” used in queries (forja.findMany('user'))
  tableName: 'users',     // Optional ā€” defaults to pluralized name ('user' ā†’ 'users')
  fields: { ... },
  indexes: [ ... ],       // Optional ā€” database indexes
}
```

### `tableName`

By default Forja pluralizes the schema name for the table name (`user` ā†’ `users`,
`category` ā†’ `categories`). Override with `tableName` when needed:

```typescript
{
  name: 'userProfile',
  tableName: 'user_profiles',  // explicit table name
  fields: { ... },
}
```

## Field types

### `string`

```typescript
name: {
  type: 'string',
  required: true,
  minLength: 2,
  maxLength: 100,
  unique: false,
  pattern: /^[a-zA-Z\s]+$/,
  default: '',
  validator: (value) => value.trim().length > 0 || 'Name cannot be blank',
}
```

| Option | Type | Description |
|--------|------|-------------|
| `required` | `boolean` | Field must be present on create |
| `minLength` | `number` | Minimum character count |
| `maxLength` | `number` | Maximum character count |
| `unique` | `boolean` | Enforces unique constraint in DB |
| `pattern` | `RegExp` | Value must match regex |
| `default` | `string` | Default value when not provided |
| `validator` | `(v) => true \| string` | Custom validation ā€” return `true` or error message |

### `number`

```typescript
age: {
  type: 'number',
  required: false,
  min: 0,
  max: 120,
  integer: true,
  unique: false,
  default: 0,
  validator: (value) => value % 2 === 0 || 'Must be even',
}
```

| Option | Type | Description |
|--------|------|-------------|
| `min` | `number` | Minimum value |
| `max` | `number` | Maximum value |
| `integer` | `boolean` | Must be a whole number |
| `unique` | `boolean` | Enforces unique constraint |
| `autoIncrement` | `boolean` | Auto-increment (used internally for `id`) |

### `boolean`

```typescript
isActive: {
  type: 'boolean',
  required: false,
  default: true,
}
```

### `date`

```typescript
publishedAt: {
  type: 'date',
  required: false,
  min: new Date('2020-01-01'),
  max: new Date(),
}
```

| Option | Type | Description |
|--------|------|-------------|
| `min` | `Date` | Earliest allowed date |
| `max` | `Date` | Latest allowed date |

### `enum`

```typescript
role: {
  type: 'enum',
  values: ['admin', 'editor', 'user'],
  required: true,
  default: 'user',
}
```

`values` is the exhaustive list of allowed strings. Any other value fails validation.

### `array`

```typescript
tags: {
  type: 'array',
  items: { type: 'string' },
  minItems: 1,
  maxItems: 10,
  unique: true,
}
```

| Option | Type | Description |
|--------|------|-------------|
| `items` | field definition | Type of each item in the array |
| `minItems` | `number` | Minimum number of items |
| `maxItems` | `number` | Maximum number of items |
| `unique` | `boolean` | No duplicate values allowed |

### `json`

```typescript
metadata: {
  type: 'json',
  required: false,
}
```

Stores any valid JSON value. No deep validation is performed ā€” useful for flexible or
dynamic data structures.

### `relation`

See [Relations](#relations) below.

## Relations

Relations describe how schemas connect to each other. Forja automatically manages
foreign keys and junction tables ā€” you never define them manually.

### `belongsTo`

The current schema holds the foreign key. Use this when a record "belongs to" another.

```typescript
// post belongs to user
author: {
  type: 'relation',
  kind: 'belongsTo',
  model: 'user',
  required: true,
  // Forja adds `authorId` foreign key column automatically
}
```

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `model` | `string` | ā€” | Target schema name |
| `required` | `boolean` | `false` | Whether the relation is required |
| `foreignKey` | `string` | `${fieldName}Id` | Custom FK column name |
| `onDelete` | `'cascade' \| 'setNull' \| 'restrict'` | `'setNull'` (`'cascade'` if required) | FK constraint behavior |
| `onUpdate` | `'cascade' \| 'restrict'` | ā€” | FK update constraint |

### `hasOne`

The target schema holds the foreign key. One-to-one from the source side.

```typescript
// user hasOne profile
profile: {
  type: 'relation',
  kind: 'hasOne',
  model: 'profile',
  // Forja adds `userId` to the profile schema automatically
}
```

### `hasMany`

One-to-many. The target schema holds the foreign key.

```typescript
// user hasMany posts
posts: {
  type: 'relation',
  kind: 'hasMany',
  model: 'post',
  // Forja adds `userId` to the post schema automatically
}
```

### `manyToMany`

Many-to-many. Forja creates a junction table automatically.

```typescript
// post manyToMany tags
tags: {
  type: 'relation',
  kind: 'manyToMany',
  model: 'tag',
  // Forja creates a `post_tag` junction table automatically
  // Override with: through: 'custom_junction_table'
}
```

## Indexes

```typescript
indexes: [
  { fields: ['email'], unique: true },
  { fields: ['createdAt'] },
  { fields: ['firstName', 'lastName'], unique: false },
]
```

## How Forja initializes schemas

When you call `getForja()` for the first time, Forja runs the following sequence:

1. **Register internal metadata schema** ā€” `_forja` table for internal bookkeeping
2. **Register user schemas** ā€” all schemas from your config
3. **Register plugin schemas** ā€” plugins can contribute their own schemas (e.g. soft-delete adds `deletedAt`)
4. **Apply schema extensions** ā€” plugins can add/remove/modify fields on existing schemas
5. **Register migration schema** ā€” `_forja_migration` table for migration history
6. **Finalize registry** ā€” processes all relations (adds FK columns, creates junction tables), sorts schemas by dependency order
7. **Initialize plugins** ā€” plugins receive the final schema state
8. **Dispatch schema load event** ā€” plugins can react to the finalized schema registry

This order means:
- Plugins can extend user schemas before relations are processed
- Foreign keys and junction tables are always generated from the final schema state
- You never need to define FK columns manually ā€” Forja handles them

## Schema registry

After initialization, you can inspect registered schemas at runtime:

```typescript
const forja = await getForja()

// Get a single schema
const schema = forja.getSchema('user')

// Get all schemas
const schemas = forja.getAllSchemas()

// Check if schema exists
const exists = forja.hasSchema('user')
```
